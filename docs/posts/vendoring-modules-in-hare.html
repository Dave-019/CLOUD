Vendoring Modules in Hare

Hare doesn't have a package manager. This is by design and I'm glad for it. No dependency hell, no lock files, no node_modules black hole. However, you still need a way to pull in third-party code. There are two approaches:

  1. system-wide installation via make install, and
  2. vendoring directly into your project with git subtree.

I'll use hare-json as the running example throughout. Our goal is to be able to use this module like so:

use encoding::json;
use fmt

export fn main() void = {
	let obj = json::newobject();
	defer json::finish(obj);

	json::set(&obj, "hello", "world")!;
	json::set(&obj, "foo", "bar")!;
	json::set(&obj, "the answer", 42.0)!;
	...
};

The encoding/ folder within the hare-json module needs to be found by the Hare toolchain so that when we use it it will then follow the folder path encoding/json/ via encoding::json and find all the *.ha files.

How Hare finds modules

When you write use encoding::json; in your code, the build driver looks for a directory called encoding/json/ containing .ha files. It searches a list of source roots in this order:

  1. The current working directory (.) which is always first, always implicit
  2. Each directory listed in the HAREPATH environment variable, colon-delimited
  3. If HAREPATH is unset, a compiled-in default (typically the stdlib and third-party install locations)

Check yours with:

$ hare version -v
hare 0.26.0

        +x86_64
        +linux
tool paths:
HAREPATH (from environment):
        vendor/*/
        /usr/src/hare/third-party
        /usr/src/hare/stdlib
        /usr/local/src/hare/third-party
        /usr/local/src/hare/stdlib
toolchains:
 aarch64:
        AS=as
        CC=cc
        LD=ld
 riscv64:
        AS=as
        CC=cc
        LD=ld

        AS=as
        CC=cc

This is the result of mine since I set my HAREPATH in my .zshrc, but on a typical Linux system with Hare installed to /usr/local, the default HAREPATH looks something like:

/usr/local/src/hare/stdlib:/usr/local/src/hare/third-party

First directory with .ha files matching the module path wins. That's the entire resolution algorithm. Simple.

The repo layout convention

Most Hare libraries follow a naming convention: the repo is called hare-MODULE, and the actual module code lives in a top-level directory named after the module. For hare-json:

hare-json/
├── encoding/
│   └── json/
│       ├── types.ha
│       ├── lex.ha
│       ├── value.ha
│       └── ...
├── Makefile
├── README
└── COPYING

The code you import as encoding::json lives under encoding/json/. This means the parent directory of encoding/ is what needs to be on HAREPATH. Not the json/ directory itself. Not the repo root unless the repo root is that parent directory.

This is where most people trip up.

Approach 1: System-wide install

Clone, make, done.

$ git clone https://git.sr.ht/~sircmpwn/hare-json
$ cd hare-json
$ make install

The Makefile handles putting .ha files into the right location. A typical Hare library Makefile looks like:

.POSIX:
.SUFFIXES:

HARE=hare
HAREFLAGS=
DESTDIR=
PREFIX=/usr/local
SRCDIR=$(PREFIX)/src
HARESRCDIR=$(SRCDIR)/hare
THIRDPARTYDIR=$(HARESRCDIR)/third-party

install:
	install -Dm644 encoding/json/* $(DESTDIR)$(THIRDPARTYDIR)/encoding/json/

uninstall:
	rm -rf $(DESTDIR)$(THIRDPARTYDIR)/encoding/json

The default THIRDPARTYDIR resolves to /usr/local/src/hare/third-party, which is already in the default HAREPATH. After make install, use encoding::json; just works.

If your Hare installation lives under /usr instead of /usr/local:

$ make PREFIX=/usr install

Or a custom path entirely:

$ make THIRDPARTYDIR=/opt/hare/libs install

Just make sure whatever path you install to is in your HAREPATH.

This approach is clean and simple. Every project on the system can use the module with zero configuration. The downside is versioning: all projects share the same installed version. If project A needs an older hare-json and project B needs a newer one, you're stuck. There's no record in your project's repo of which dependencies you need or what versions they are either. A new contributor has to read your README and manually install each one.

Approach 2: Vendoring with git subtree

For reproducible builds and per-project dependency management, vendor the module source directly into your project.

From your project root:

$ git subtree add --prefix vendor/hare-json \
    https://git.sr.ht/~sircmpwn/hare-json master --squash

This creates vendor/hare-json/ with the full contents of the repo:

my-project/
├── main.ha
├── vendor/
│   └── hare-json/
│       ├── encoding/
│       │   └── json/
│       │       ├── types.ha
│       │       ├── lex.ha
│       │       └── ...
│       ├── Makefile
│       └── README
└── Makefile

Remember the layout convention: encoding/json/ is relative to the repo root vendor/hare-json/. So the path that goes on HAREPATH is vendor/hare-json. Not vendor/. Not vendor/hare-json/encoding/json/.

Updating a vendored dependency

$ git subtree pull --prefix vendor/hare-json \
    https://git.sr.ht/~sircmpwn/hare-json master --squash

Upstream changes get merged in. --squash collapses the upstream history into a single commit.

Setting HAREPATH

Important: when you set HAREPATH as an environment variable, it completely replaces the compiled-in default. You must re-include the stdlib and third-party paths yourself or the build driver won't find the standard library. Run hare version -v to see what those paths are on your system.

The Hare docs mention establishing vendor/*/ as a default convention for HAREPATH, but as of this writing it does not work. The glob does not expand the way you'd expect and the build driver won't find your modules. You need to specify each vendored module path explicitly.

The way to do this is in your project Makefile:

.POSIX:
.SUFFIXES:

HARE=hare
HAREFLAGS=

HAREPATH=vendor/hare-json:/usr/local/src/hare/third-party:/usr/local/src/hare/stdlib

build:
	HAREPATH=$(HAREPATH) $(HARE) build $(HAREFLAGS) -o my-project cmd/my-project/

check:
	HAREPATH=$(HAREPATH) $(HARE) test $(HAREFLAGS)

Multiple vendored deps, list each one:

HAREPATH=vendor/hare-json:vendor/hare-xml:vendor/hare-http:/usr/local/src/hare/third-party:/usr/local/src/hare/stdlib

Colon-separated, just like PATH. Adjust the stdlib and third-party paths to match your system. The tradeoff is you need to update the Makefile every time you add or remove a vendored dependency, and the stdlib paths are hardcoded to one install prefix.

Vendoring pros and cons

You get reproducible builds, per-project versioning, and a self-contained repo that any contributor can clone and build immediately. Dependencies are explicit in the source tree.

The cost is a larger repo since you're committing the dependency source and a slightly more complex build setup. You also need to manually pull upstream changes when you want them.

Which to use

Personal projects, scripts, small tools: make install. Fast, simple, done.

Anything collaborative, anything you want reproducible builds for, anything that might need pinned dependency versions: vendor with git subtree. The overhead is minimal and the payoff is real.

You can combine both. Install common modules system-wide for convenience during development and vendor them for CI and release builds. Since . has the highest priority in HAREPATH and vendored paths come before the defaults, the resolution order does what you'd expect.
